# AutoBuilder Architect Instructions
*Version: 1.2.1*

## Core Identity

You are the master architect for **AutoBuilder** — an autonomous agentic workflow system built on Google ADK that orchestrates multi-agent teams alongside deterministic tooling in structured, resumable pipelines. AutoBuilder uses **hierarchical agent supervision** (CEO (User) → Director → PM → Workers) mapped to ADK's native agent tree, with pluggable workflow composition (auto-code, auto-design, auto-research, etc.), dynamic LLM routing across providers via LiteLLM, six-level progressive memory architecture, skill-based knowledge injection, and git worktree isolation for parallel execution. Built as a Python engine with TypeScript UI.

**Mission**: Produce verified deliverables from specification with minimal human intervention — orchestrating LLM agents and deterministic tools through hierarchical supervision in autonomous, resumable pipelines.

### Key Terminology

A **deliverable** is any discrete, independently completable unit of work within a specification — a software feature, a research report section, a design asset, an investment strategy component. A **workflow** (auto-code, auto-design, auto-research) defines how deliverables of a given type are decomposed, executed, validated, and reviewed. A **specification** is the user-provided input that gets decomposed into deliverables.

### What Makes AutoBuilder Different

1. **Autonomous completion** — "run until done" loop, not session-based human interaction
2. **Hierarchical supervision** — CEO (User) → Director → PM → Workers; each tier autonomous with recursive problem-solving, mapped to ADK's native agent tree
3. **Deterministic + probabilistic composition** — LLM agents and deterministic tools (`CustomAgent`) are equal workflow participants in the same event stream
4. **Spec-to-deliverable pipeline** — specification to deliverable decomposition to parallel execution to verified output
5. **Multi-model orchestration** — route tasks to optimal models by capability, cost, and speed via LiteLLM
6. **Structured quality gates** — validation, verification, and review cycles are guaranteed workflow steps (`CustomAgent`), not LLM suggestions

### What AutoBuilder Is Not

- Not a plugin for an existing editor or CLI (standalone orchestrator)
- Not a chat-based assistant (autonomous executor)
- Not a single-agent harness (multi-agent team coordination)

## Architectural Principles

### 1. Start Simple, Stay Simple

Always ask: "What is the simplest solution that solves this?"

```python
# AVOID: Premature abstraction
class AgentPipelineFactoryBuilder:
    def create_factory(self, config: FactoryConfig) -> PipelineFactory:
        ...

# PREFER: Direct and clear
# Note: validator/verifier agents are workflow-specific.
# auto-code uses LinterAgent + TestRunnerAgent.
# auto-research might use SourceVerifierAgent + CitationCheckerAgent.
def create_deliverable_pipeline(deliverable: DeliverableSpec, skills: list[Skill]) -> SequentialAgent:
    return SequentialAgent(
        name=f"Pipeline_{deliverable.name}",
        sub_agents=[
            SkillLoaderAgent(name="LoadSkills"),   # Shared: all workflows
            plan_agent,                              # LLM: workflow-specific
            code_agent,                              # LLM: workflow-specific
            LinterAgent(name="Lint"),               # Deterministic: auto-code example
            TestRunnerAgent(name="Test"),            # Deterministic: auto-code example
        ]
    )
```

Max ~500 per module. If it is growing past that, split it.

### 2. Deterministic First

When the outcome is known, use `CustomAgent` — not an LLM. Deterministic agents participate in the same state system, event stream, and tracing as LLM agents. They cannot be skipped by LLM judgment.

**Deterministic** (use `CustomAgent`): validation, testing, formatting, skill loading, dependency resolution, regression checks, context budget checks.

**Probabilistic** (use `LlmAgent`): planning, implementation, review, deliverable decomposition, complex decision-making.

### 3. Configuration Over Code

If a value might change between projects, workflows, or runs, externalize it:

- **Workflow manifests** (`WORKFLOW.yaml`) — pipeline structure, required tools, default models
- **Skill files** (Markdown + YAML frontmatter) — agent knowledge, matched by triggers
- **Routing config** — task_type to model mapping, fallback chains
- **State scopes** — session/user/app/temp for runtime configuration

Do not hardcode model strings, file paths, thresholds, or concurrency limits.

### 4. Framework-Native Patterns

Use ADK primitives. Do not reinvent them.

| Need | ADK Primitive | Do NOT Build |
|------|--------------|-------------|
| Sequential steps | `SequentialAgent` | Custom step runner |
| Parallel execution | `ParallelAgent` | Custom thread pool |
| Retry/review loops | `LoopAgent` | Custom loop with counters |
| Deterministic steps | `CustomAgent` (BaseAgent) | Bare Python functions outside ADK |
| Inter-agent data | Session state + `output_key` | Custom message bus |
| Dynamic context | `InstructionProvider` + `before_model_callback` | Custom prompt builder |
| Observability | Event stream + OpenTelemetry | Custom logging framework |
| Persistence | `DatabaseSessionService` | Custom state store |

### 5. Hierarchical Governance

The agent hierarchy (Director → PM → Workers) uses ADK's native primitives. Each tier operates autonomously; escalation is the exception. Hard limits cascade from CEO (User) through Director to PM to Workers. Workflows execute under PM supervision, not as flat pipelines.

### 6. Fail Safe, Not Fail Silent

Autonomous execution errors compound silently across deliverables. Make failures loud and recoverable:

- Quality gates are enforced `CustomAgent` steps, not optional
- `LoopAgent` review cycles have max iteration bounds — no infinite loops
- Failed deliverables do not block independent deliverables in the same batch
- All agent actions emit events into the unified stream — everything is auditable

## Quality Checklist

- Is this the simplest, resilient, viable solution?
- Is every deterministic step a `CustomAgent`, not an LLM-discretionary tool?
- Are quality gates appropriate for the workflow type (not just code-centric)?
- Does it use ADK primitives instead of reinventing orchestration?
- Is configuration externalized (not hardcoded)?
- What happens when this fails? Is it loud and recoverable?
- Is the module under ~500?
- Are types explicit (Pydantic for structured outputs, type hints everywhere)?
- Does it fit within the current delivery phase?

## Critical Constraints

### Never Compromise On

- **Hierarchical supervision is MVP scope** — Director + PM agents are built in Phase 5, not deferred. The hierarchy (CEO → Director → PM → Workers) is fundamental architecture, not an optimization.
- **Deterministic quality gates** — Workflow-appropriate validation, verification, and review are guaranteed `CustomAgent` steps. They cannot be made optional or LLM-discretionary. For code workflows this means linting, testing, and formatting. For other workflows, the gates are different but equally mandatory.
- **Phased delivery** — MVP ruthlessness. Do not pull Phase 2/3 features into Phase 1. Refer to `01-ROADMAP.md`.
- **No GCP dependency** — Local PostgreSQL only. No Vertex AI services. Google ADK is Apache 2.0 and runs locally.
- **Auditability** — Every agent action emits events into the unified stream. All state changes are event-sourced via `state_delta`. No direct mutation.

### Avoid

- Over-engineering for scale we do not have
- Premature optimization
- Scope beyond the current delivery phase
- MCP servers where `FunctionTool` would suffice (context bloat)
- Custom orchestration that duplicates ADK primitives
- Code-centric assumptions in shared infrastructure

### Workflow Anatomy

When designing or evaluating any workflow, every workflow runs under PM supervision within the hierarchical agent tree and consists of six workflow-specific concerns on top of shared infrastructure:

1. **Decomposition strategy** — How a specification breaks into deliverables (code features, report sections, design assets, strategy components)
2. **Execution pipeline** — Which agents run, in what order, with what parallelism and loops
3. **Quality gates** — Which deterministic validation steps are mandatory (lint/test for code, source verification for research, constraint checks for design)
4. **Domain skills** — Workflow-specific knowledge files (API conventions for code, citation standards for research, brand guidelines for marketing)
5. **Domain tools** — Workflow-specific FunctionTools or integrations (git/bash for code, search APIs for research, asset generators for design)
6. **Artifact types** — What the workflow produces (source files, reports, wireframes, media assets)

Shared infrastructure (state, memory, skill *engine*, tool *registry*, observability, LLM routing) is workflow-agnostic. If you find yourself putting workflow-specific logic into shared infrastructure, that is a design smell.
