# AutoBuilder Engineering Standards
*Version: 1.0.0*

## Philosophy

**"Start simple, stay simple"** — Complexity is only added to solve proven problems, not potential ones.

**"Deterministic where possible, probabilistic where necessary"** — When the outcome is known (lint, test, format, route), use a `CustomAgent` or plain function. Reserve LLM agents for tasks that genuinely require judgment.

## STOP OVER-ENGINEERING

**You are over-engineering if:**

1. Creating abstractions for fewer than 3 real implementations that exist TODAY
2. Optimizing for edge cases (5%) over common cases (95%)
3. Adding indirection "just in case"
4. Building for 1M users when we have 10

**DEFAULT: Simple functions and objects.**

## A. Core Principles

### 1. Simplicity is Elegant
- "What is the simplest, resilient model that solves the UNIFIED ARCHITECTURE need?"
- Start simple, optimize only when measured
- Do not let 5% edge cases complicate 95% of code
- Max ~500 lines per module — if a file grows past this, split it

### 2. Deterministic > Probabilistic
- When the outcome is known, use `CustomAgent` (deterministic), not `LlmAgent`
- Linting, testing, formatting, skill loading, dependency resolution — these are guaranteed steps, not LLM suggestions
- `CustomAgent` subclasses participate in ADK's event stream, state system, and tracing identically to LLM agents
- Reserve LLM agents for planning, coding, reviewing — tasks requiring genuine judgment

### 3. Single Source of Truth
- Define once, reference everywhere
- Abstract only after third occurrence
- **State scopes** are the source of truth for runtime data (session/user/app/temp)
- **Skill files** are the source of truth for agent knowledge (Markdown + YAML frontmatter)
- **Workflow manifests** (`WORKFLOW.yaml`) are the source of truth for pipeline configuration
- **Routing config** is the source of truth for model selection

### 4. Fail Safe, Not Silent
- Autonomous execution errors can compound silently across features
- Make failures loud and recoverable
- Quality gates (lint, test, review) are enforced workflow steps, not optional suggestions
- All decisions auditable via ADK's event stream

### 5. Async by Default
- All I/O operations asynchronous (`asyncio` throughout)
- `async def` for agent implementations, tool functions, and service calls
- Use `asyncio.gather` or ADK's `ParallelAgent` for concurrent execution
- `pytest-asyncio` for async test execution

### 6. Max ~300 Lines Per Module
- If a module exceeds ~300 lines, it is doing too much — split it
- This applies to agent definitions, tools, services, and utilities
- Exceptions are rare and must be documented

## B. Implementation Rules

### Avoid
- Abstractions for single use cases
- Factories with fewer than 3 implementations
- `Any` type — use explicit types or `object` for truly dynamic data
- Hardcoded configuration values (model strings, thresholds, limits)
- Inheritance deeper than 2 levels
- MCP servers where a `FunctionTool` would suffice (MCPs add context bloat)

### Prefer
- Direct function calls over indirection
- `Protocol` classes for multi-implementation interfaces
- Explicit code over clever abstractions
- `FunctionTool` wrappers over MCP servers (lighter token footprint)
- `CustomAgent` for guaranteed workflow steps over LLM-discretionary tool calls
- Composition (`SequentialAgent`, `ParallelAgent`, `LoopAgent`) over custom orchestration

### Type Safety
- **Pydantic models** for structured outputs from LLM agents and external boundaries
- **Type hints everywhere** — every function signature, every variable where non-obvious
- **pyright strict mode** enforced
- **`dataclass`** or **`TypedDict`** for internal data transfer objects
- Native types internally: `datetime`, `Path`, `Enum`
- Conversion only at boundaries (API input/output, file I/O, LLM responses)

### Error Handling
- Catch blocks use `unknown` (or `Exception` with narrowing)
- Narrow with `isinstance` checks or guard functions
- Return sensible defaults rather than throwing where possible
- Always handle async errors — no unhandled promise rejections or unhandled exceptions in agents
- Document exceptions a function can raise

```python
# Correct: catch unknown, narrow explicitly
try:
    result = await agent.run_async(ctx)
except Exception as e:
    if isinstance(e, RateLimitError):
        await backoff_and_retry(e)
    elif isinstance(e, ToolExecutionError):
        logger.error(f"Tool failed: {e.tool_name} — {e.message}")
        return ToolResult(success=False, error=str(e))
    else:
        logger.error(f"Unexpected error: {e}")
        raise
```

## C. Naming Conventions

| Kind | Convention | Example |
|------|-----------|---------|
| Files | `snake_case.py` | `skill_loader.py` |
| Functions | `snake_case` | `resolve_model_string()` |
| Classes | `PascalCase` | `BatchOrchestrator` |
| Interfaces (Protocol) | `PascalCase` | `MemoryService` |
| Constants | `SCREAMING_SNAKE` | `MAX_CONCURRENCY`, `DEFAULT_MODEL` |
| Pydantic models | `PascalCase` | `RoutingConfig`, `FeatureSpec` |
| Agent names (ADK) | `PascalCase` string | `name="PlanAgent"`, `name="LinterAgent"` |
| State keys | `snake_case` with scope prefix | `current_feature_spec`, `app:coding_standards` |
| Test files | `test_<module>.py` | `test_skill_loader.py` |

## D. Testing

- **Framework**: `pytest` with `pytest-asyncio` for async tests
- **Factory patterns** for test data: `create_feature()`, `create_session()`, `create_skill_entry()`
- **Coverage target**: >80% line coverage
- **Test files**: `test_<module>.py` inside a `tests/` directory mirroring source structure
- Cover edge cases: empty inputs, missing state keys, agent failures, timeout scenarios
- Mock external services (LLM calls, filesystem) — never call real APIs in unit tests
- Integration tests may use real ADK primitives with `InMemorySessionService`

```python
# Factory pattern example
def create_feature(
    name: str = "test-feature",
    feature_type: str = "api_endpoint",
    status: str = "pending",
) -> FeatureSpec:
    return FeatureSpec(name=name, feature_type=feature_type, status=status)
```

## E. Code Quality

- **Linting + Formatting**: `ruff` (single tool for both)
  - Line length: 100
  - Quote style: TBD (double or single — decide before first commit)
  - Import sorting: isort-compatible rules via ruff
- **Type checking**: `pyright` in strict mode
- **Pre-commit hooks**: ruff lint, ruff format, pyright check
- All three must pass before any commit is accepted

## F. Quick Reference Decision Checklist

- What is the simplest solution that works?
- Is this deterministic or does it genuinely need an LLM?
- Do I have 3+ real implementations TODAY to justify this abstraction?
- What happens when this fails? Is the failure visible and recoverable?
- Is this under ~300 lines?
- Are types explicit? Would pyright pass in strict mode?
- Is configuration externalized (not hardcoded)?

### Keywords
`SIMPLE` | `DETERMINISTIC` | `DRY` | `ASYNC` | `AUDITABLE` | `TYPED`

## G. Exceptions

When breaking standards, document why:

```python
# EXCEPTION: Synchronous call — this library does not support async
# Review: Phase 2 — evaluate async alternative
```
