# Development Workflow

Four levels from idea to verified code. Each level has a distinct owner and abstraction. Upstream is aspirational, downstream is testable. Changes propagate bidirectionally — downstream layers need more upstream knowledge than the reverse.

No artifact at any level contains implementation code. Code is produced exclusively by Builder Agents in Level 4b.

```
1. SHAPE       Why + What (product)
2. DESIGN      How (system)
3. PLAN        When + Where (phases)
4. BUILD       Do (per phase)
   4a. Plan    FRD → Spec + Model
   4b. Build   Implement → Test → Review → Iterate
```

### Document Numbering

Documents are numbered in workflow order — the sequence you'd read them to understand the project top-down.

| # | Document | Level | Purpose |
|---|----------|-------|---------|
| 00 | `00-VISION.md` | Shape | Mission, innovation, north star |
| 01 | `01-PRD.md` | Shape | Product requirements — user experience deconstructed |
| 02 | `02-ARCHITECTURE.md` | Design | L1 system architecture overview |

## Level 1: SHAPE — Product Definition

**Owner**: Founder / Product
**When**: Project inception; revisited when mission or product scope changes.

Two artifacts that answer "why does this exist?" and "what must it do?"

### 1a. Vision & Strategy

**Artifact**: `00-VISION.md`
**Audience**: Everyone — team, investors, yourself at 3am.

The aspirational document. If this product were at its 10th iteration, fully evolved, what would it look like? No implementation details — this is the dream.

| Section | What It Answers |
|---------|-----------------|
| **Mission** | One sentence. Why does this product exist? What change does it create? |
| **Problem** | What pain exists today? Who feels it? Why do current solutions fail? Name the frustrations. |
| **Who It's For** | Target users as real scenarios, not demographics. |
| **Product Vision** | The fully-realized product at maturity. What can users do? What does it feel like? |
| **Innovations** | What is genuinely new? Not "we use AI" — what specific breakthroughs or novel combinations? |
| **Strategic Advantages** | Moat. Why is this hard to replicate? What compounds over time? |
| **Competitive Landscape** | What exists, why it's insufficient. Honest about alternatives' strengths. |
| **Non-Goals** | What this product will never be. Prevents mission drift. |
| **Success Criteria** | How you know you've won. Concrete outcomes, not vanity metrics. |

**Exit**: A new team member reads this and understands *why* the product exists, *what's unique*, and *what winning looks like* — without any technical documents.

### 1b. Product Requirements Document (PRD)

**Artifact**: `01-PRD.md`
**Audience**: Engineering, design, QA.
**Maps to**: Vision — every requirement traces to a vision goal.

Takes the vision and deconstructs it into everything the user must experience. The bridge between dream and engineering. No implementation — that's Design's job.

| Section | What It Answers |
|---------|-----------------|
| **User Personas** | 2-4 behavioral archetypes with goals, frustrations, and context. |
| **User Journeys** | End-to-end flows per persona. Trigger → actions → outcome. Critical paths. |
| **Functional Requirements** | What the product must do, by domain area. "When X, the system Y." Prioritized. |
| **Non-Functional Requirements** | Performance, reliability, security, scalability, accessibility — product-wide. |
| **Information Architecture** | Core entities, relationships. What the user sees, creates, modifies, deletes. |
| **Integration Requirements** | External systems, protocols, data formats. |
| **Constraints & Assumptions** | Business/technical constraints. Assumptions that if wrong would change requirements. |
| **Out of Scope** | Explicitly deferred or excluded. "Not yet" vs "never." |
| **Success Metrics** | Product-level KPIs tied to user outcomes. |
| **Traceability** | Map table: every requirement → vision goal it serves. |

**Exit**: An engineer reads this and knows *what the product does from the user's perspective* without assumptions. Every vision goal traces to at least one requirement.

## Level 2: DESIGN — System Architecture

**Owner**: Lead Engineer / Architect
**When**: After PRD defines what to build. L1 first, L2 iterates as planning progresses.

Translates product requirements into system design. No implementation code — models, interfaces, contracts, and design decisions only.

### Artifacts

| Artifact | Path | Purpose |
|----------|------|---------|
| **L1 Architecture** | `02-ARCHITECTURE.md` | System overview — layers, data flow, integration boundaries, key patterns |
| **L2 Architecture** | `architecture/*.md` | Domain-specific sub-system architecture — components, interfaces, protocols, state |
| Project Structure | `03-STRUCTURE.md` | Directory layout (single source of truth for file placement) |
| Tech Stack | `04-TECH_STACK.md` | Every technology choice with rationale and rejected alternatives |
| Standards | `.standards.md` | Code style, patterns, conventions, anti-patterns |
| Dev Setup | `05-DEV_SETUP.md` | Local development environment |
| Providers | `06-PROVIDERS.md` | External services, models, pricing, fallback chains |

**Maps to**: PRD — every architectural decision serves a product requirement.

L1 is the top-level product architecture overview. L2 files are domain-specific sub-system architecture that define the components, interfaces, and contracts each phase must conform to. L2 is a direct input to the Build-level Spec and Model.

| Required Map (in `02-ARCHITECTURE.md`) | |
|---|---|
| PRD Requirement | Architecture Component(s) |
| *Every PRD functional requirement must trace to at least one architecture component.* |

**Exit**: Every PRD requirement has a home in the architecture. Every component identifiable. File placement defined.

## Level 3: PLAN — Phased Delivery

**Owner**: Engineering Lead
**When**: After L1 is stable and L2 has enough detail to identify components.

Decomposes the architecture into buildable phases. Every component gets assigned. Every phase gets a completion contract.

### Artifacts

| Artifact | Path | Maps To |
|----------|------|---------|
| **Component Registry (BOM)** | `07-COMPONENTS.md` | L2 Architecture — every component derived from an architecture domain file |
| **Roadmap** | `08-ROADMAP.md` | BOM (phases group components) + PRD (contracts trace to product requirements) |

| Required Map (in `07-COMPONENTS.md`) | |
|---|---|
| Component | L2 Architecture Source |
| *Every component traces to an architecture domain file section.* |

| Required Map (in `08-ROADMAP.md`) | |
|---|---|
| Completion Contract Item | PRD Requirement(s) |
| *Every contract item traces to the product requirement it satisfies.* |

**Exit**: Every architecture component assigned to a phase (zero gaps). Every phase has a testable completion contract. Every contract traces to PRD.

## Level 4: BUILD — Phase Execution

**Owner**: Engineering
**When**: Per phase, in roadmap order.

Two sub-pipelines: Plan the phase, then build it. No implementation code until 4b.

### 4a. Phase Plan: FRD → Spec + Model

Three artifacts. FRD first, then Spec and Model in parallel (Spec derives from FRD + BOM + Roadmap; Model derives from FRD + L2 Architecture; neither depends on the other).

#### FRD — Functional Requirements Document

**Command**: `/shape-phase {N}`
**Artifact**: `.dev/build-phase/phase-{N}/frd.md`
**Template**: `.dev/build-phase/.templates/frd.md`
**Maps to**: PRD (phase requirements derive from product requirements) + Roadmap contract items

Deconstructs the phase into capabilities and testable requirements. What must this phase do, and how do I prove it works?

**Capability vs Functional Requirement:**
A **Capability (CAP-{n})** is a named consumer-facing behavior group — the container. A **Functional Requirement (FR-{N}.{nn})** is a specific, testable condition *within* a capability — the proof. Multiple FRs live under each capability. All FRs under a capability must pass for the capability to be satisfied.

- **Capabilities** grouped by consumer-facing behavior
- **Functional requirements** with IDs — testable behaviors, not implementation
- **Edge cases and error scenarios** per capability
- **Non-functional requirements** — performance, reliability, security, limits
- **Rabbit holes** — known gotchas identified upfront
- **No-gos** — explicit exclusions preventing scope creep

| Required Map | |
|---|---|
| FRD Requirement (FR-{N}.{nn}) | PRD Requirement |
| Roadmap Contract Item | FRD Capability (CAP-{n}) |
| *Every FRD requirement traces to a PRD requirement. Every roadmap contract traces to a capability.* |

**Exit**: Every capability has testable requirements. Every roadmap contract item traces to a capability. Every FRD requirement traces to PRD.

#### Spec — Technical Specification

**Command**: `/spec-phase {N}`
**Artifact**: `.dev/build-phase/phase-{N}/spec.md`
**Template**: `.dev/build-phase/.templates/spec.md`
**Maps to**: FRD (requirements to satisfy) + BOM (components to implement) + Roadmap (contract items to cover)

Decomposes FRD requirements into buildable deliverables — files, dependencies, build order, validation commands. No implementation code — describes *what* to build and *in what order*, not *how*.

| Required Map | |
|---|---|
| FRD Requirement (FR-{N}.{nn}) | Deliverable (P{N}.D{n}) |
| BOM Component | Deliverable (P{N}.D{n}) |
| Roadmap Contract Item | Deliverable (P{N}.D{n}) |
| *Every FRD requirement traces to a deliverable. Every BOM component maps to a deliverable. Every roadmap contract item traces to a deliverable. Zero unmapped.* |

**Exit**: Full traceability from FRD requirements, BOM components, and roadmap contract items to deliverables. Build order defined. Validation commands specified.

#### Model — Phase Architecture Model

**Command**: `/model-phase {N}`
**Artifact**: `.dev/build-phase/phase-{N}/model.md`
**Template**: `.dev/build-phase/.templates/model.md`
**Maps to**: FRD (requirements to model) + L2 Architecture (conformance target)

Defines the architecture contracts (interfaces, types, data flow) the build must conform to. No implementation code — Protocol/ABC signatures with `...` bodies only. Optional for simple phases.

| Required Map | |
|---|---|
| Model Component | L2 Architecture Section |
| *Every model component conforms to an L2 architecture section.* |

**Exit**: Interfaces defined. Data flow diagrammed. Every model component traces to its L2 architecture section.

### 4b. Phase Build: Implement → Test → Review → Iterate

**Command**: `/build-phase {N}`
**Artifact**: Verified code + `.dev/build-phase/phase-{N}/review.md`
**Inputs**: Spec (deliverables), Model (contracts), FRD (requirements to verify against), Standards

This is where code gets written — by Builder Agents, not by upstream artifacts.

Five steps:
1. **Plan** — Map deliverables to existing code, confirm build order
2. **Implement** — Build batch-by-batch per spec, validate per deliverable
3. **Quality Gate** — Lint + types + tests, zero errors
4. **Review Gate** — Code review (configurable: double/single/none), fix all findings
5. **Completion Protocol** — Verify every contract item with evidence, update roadmap

**Verification Layers** — three independent layers, all required:

| Layer | Question | Source | Can fail independently? |
|-------|----------|--------|------------------------|
| **FRs** | Did it work correctly? | `frd.md` FR-{N}.{nn} | Yes — right files, wrong behavior |
| **Deliverables** | Was it built right? | `spec.md` BOM + requirements | Yes — passes FRs, wrong architecture |
| **Contract items** | Was the phase scope completed? | `08-ROADMAP.md` completion contract | Yes — built correctly, missed a commitment |

Passing FRs does not guarantee deliverable conformance. Checking off deliverables does not guarantee contract completion. All three must be verified with evidence.

**Exit**: All deliverables implemented. Quality gate clean. Review complete. Every FR, deliverable, and contract item verified with evidence. Roadmap updated to DONE.

## Sync Rules

Upstream changes cascade down. Downstream discoveries propagate up.

| Change In | Propagates To | Action |
|-----------|---------------|--------|
| Vision | PRD → everything downstream | Reassess product requirements against new mission |
| PRD | Architecture → BOM → Roadmap → affected FRDs | Verify architecture still serves requirements |
| L2 Architecture | BOM → Roadmap → affected Specs/Models | Re-derive components, reassign phases |
| BOM | Roadmap → affected Specs | Re-check phase assignments and deliverable mappings |
| Roadmap | Affected phase FRDs | Re-check phase scope |
| FRD | Spec + Model | Re-run `/spec-phase --resume`, re-run `/model-phase` |
| Spec/Model (discovery) | FRD → Roadmap (if scope change) | Flag upstream when build assumptions break; update FRD requirements |
