# Events in Agent Development Kit

## Overview

Events serve as the fundamental units of information flow within the ADK. They represent "every significant occurrence during an agent's interaction lifecycle, from initial user input to the final response."

## Core Structure

An `Event` object extends a basic response structure with ADK-specific metadata:

- **author**: Identifies the source ('user' or agent name)
- **invocation_id**: Tracks the entire interaction cycle
- **id**: Unique identifier for each event
- **timestamp**: Creation time
- **actions**: Signals for side effects and control flow
- **content**: The main payload (text, function calls, or results)

## Key Event Types

**User Input**: Initial messages with author='user'

**Agent Responses**: Text replies authored by specific agents

**Tool Calls**: Function call requests from the LLM

**Tool Results**: Responses from executed tools with function_response content

**State Updates**: Changes signaled via event.actions.state_delta

**Control Signals**: Commands like transfer_to_agent or escalate flags

## Identifying Events

Check these attributes:
- **event.author**: Who generated it
- **event.content.parts**: What type of payload
- **event.partial**: Whether streaming continues
- **event.actions**: What side effects occurred

## Final Response Detection

Use the built-in `event.is_final_response()` method to identify complete, user-facing outputs. Returns true when:
- Tool result has skip_summarization enabled
- Long-running tool call detected
- Message complete without function calls/responses and not partial

## Processing Flow

1. Event is generated by source (user, agent, or LLM)
2. Runner receives and processes it
3. SessionService applies state/artifact deltas
4. Event appended to session history
5. Framework yields processed event to application

## Best Practices

- Set clear authorship in custom agents
- Use content for messages, actions for side effects
- Leverage `is_final_response()` for UI display filtering
- Examine event sequences in session history for debugging
- Use invocation_id for correlation across interactions
